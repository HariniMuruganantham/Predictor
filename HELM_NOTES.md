\# Helm Chart Scaffold Comparison



\## Objective



Create the initial Helm chart scaffold for the Predictor project and compare the generated Kubernetes resources with the existing handwritten manifests.



\---



\## Tasks Completed



\* Created the Helm chart scaffold using:



```bash

helm create predictor-chart

```



\* Removed unused scaffold templates:



&#x20; \* `templates/hpa.yaml`

&#x20; \* `templates/ingress.yaml`

&#x20; \* `templates/tests/`



\* Reviewed the generated templates:



&#x20; \* `templates/deployment.yaml`

&#x20; \* `templates/service.yaml`



\* Rendered the chart using:



```bash

helm template predictor predictor-chart

```



\---



\# Structural Differences



\## Resource Naming



\*\*Helm\*\*



\* Generates resource names dynamically using the release and chart names.



Example:



```yaml

name: predictor-predictor-chart

```



\*\*Current Manifest\*\*



```yaml

name: predictor

```



\---



\## Labels



\### Helm



Uses Kubernetes recommended labels.



\* `app.kubernetes.io/name`

\* `app.kubernetes.io/instance`

\* `app.kubernetes.io/version`

\* `helm.sh/chart`



\### Current Manifest



Uses a simple application label.



```yaml

app: predictor

```



\---



\## Container Image



\### Helm



Uses configurable values from `values.yaml`.



```yaml

image:

&#x20; repository: nginx

&#x20; tag: "1.16.0"

```



\### Current Manifest



Uses the Predictor application image directly.



```yaml

image: predictor:v0.2

```



\---



\## Service Account



Helm automatically creates and assigns a ServiceAccount.



The handwritten deployment does not use a ServiceAccount.



\---



\## Image Pull Policy



\### Helm



```yaml

IfNotPresent

```



\### Current Manifest



```yaml

Never

```



\---



\## Container Port



\### Helm



```yaml

containerPort: 80

```



\### Current Manifest



```yaml

containerPort: 5000

```



\---



\## Health Probes



\### Helm



Uses the root endpoint.



```yaml

/

```



\### Current Manifest



Uses the application's health endpoint.



```yaml

/health

```



\---



\## Environment Variables



The generated Helm chart does not include ConfigMap-based environment variables.



The handwritten deployment loads `LOG\_LEVEL` from a ConfigMap.



\---



\## Resource Requests and Limits



Helm provides configurable placeholders through `values.yaml`.



The Predictor deployment defines explicit CPU and memory requests and limits.



\---



\## Service



\### Helm



Creates a `ClusterIP` Service.



\### Current Manifest



Creates a `NodePort` Service exposing the application externally.



\---



\## Configuration Style



Helm uses Go template syntax:



```yaml

{{ .Values.image.repository }}

{{ .Values.service.port }}

```



The handwritten manifests contain fixed values.



\---



\# Summary



| Feature         | Helm Scaffold              | Predictor Deployment |

| --------------- | -------------------------- | -------------------- |

| Resource names  | Dynamic                    | Static               |

| Labels          | Standard Kubernetes labels | Custom labels        |

| Image           | Configurable               | Hardcoded            |

| ServiceAccount  | Present                    | Not used             |

| Pull Policy     | IfNotPresent               | Never                |

| Container Port  | 80                         | 5000                 |

| Health Probe    | `/`                        | `/health`            |

| ConfigMap       | Not configured             | ConfigMap used       |

| Resource Limits | Placeholder values         | Explicit values      |

| Service Type    | ClusterIP                  | NodePort             |

| Configuration   | Go templates               | Static YAML          |



\---



\## Conclusion



The Helm chart generated by `helm create` is a generic application scaffold. The existing Predictor Kubernetes manifests contain application-specific configuration, including ConfigMap integration, resource limits, health probes, NodePort networking, and the Predictor container image.



The next step is to customize the Helm templates by replacing the default scaffold configuration with the Predictor application's Kubernetes configuration while moving configurable values into `values.yaml`.
